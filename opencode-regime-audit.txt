1. BUG — ABS() IN CASCADE DETECTION CAUSES FALSE POSITIVES
   File: detect_regime.py:170
   Code: abs(signals.get("price_vs_20h_high", 0)) > 1.5

   price_vs_20h_high = (btc_spot - high_24h) / high_24h * 100

   If btc_spot is ABOVE the 24h high, price_vs_20h_high is positive.
   abs() would make it > 1.5, triggering CASCADE even though price
   breaking ABOVE the high is a breakout, not a liquidation cascade.

   Fix: Change to signals.get("price_vs_20h_high", 0) < -1.5
   (price must be BELOW the high by >1.5%)

----------------------------------------------------------------

2. NAMING INCONSISTENCY — "20h" KEYS vs "24h" DATA
   File: detect_regime.py:115-119

   Data source variables: high_24h, low_24h (suggest 24-hour data)
   Output keys: price_vs_20h_high, price_vs_20h_low (say "20h")

   Either the keys should be renamed to match the data (24h), or
   the data source actually provides 20h data under misleading
   variable names. This will confuse anyone debugging regime logic.

----------------------------------------------------------------

3. MISLEADING NAME — ATR NORMALIZED IS JUST 24H RANGE %
   File: detect_regime.py:121-122

   atr_val = (high_24h - low_24h) / btc_spot * 100

   This is a simple 24h range percentage, not ATR (Average True Range).
   ATR requires True Range across multiple periods. Naming it
   "atr_normalized" misleads about what's actually being measured.

----------------------------------------------------------------

4. TIMEZONE PARSING IS FRAGILE
   File: detect_regime.py:219

   prev_timestamp.replace(" ", "T").replace(" UTC", "+00:00")

   Only works for timestamps produced by datetime.isoformat() with
   UTC timezone. Would break on:
   - "Z" suffix (ISO 8601 standard)
   - Timestamps with microseconds: "2026-01-01T12:00:00+00:00"
   - Non-UTC offsets
   - Malformed strings containing " UTC" in unexpected positions

   Consider using datetime.fromisoformat() directly on ISO timestamps,
   or adding explicit format validation.

----------------------------------------------------------------

5. NEGATIVE REGIME AGE ON CLOCK SKEW
   File: detect_regime.py:220

   regime_age_minutes = int((now - prev_ts).total_seconds() / 60)

   If system clock moves backward (NTP correction, VM migration),
   prev_ts could be in the future, producing negative age. The value
   would be displayed as a negative number in output. Not critical
   but produces confusing output.

   Fix: regime_age_minutes = max(0, int((now - prev_ts).total_seconds() / 60))

----------------------------------------------------------------

6. ZERO ATR TRIGGERS FALSE RANGING REGIME
   File: detect_regime.py:117-126, 193

   If btc_spot, high_24h, or low_24h is 0 (e.g., API returns 0 or
   feed is stale), the else branch sets atr_normalized = 0. Then:
     signals.get("atr_normalized", 999) < 0.8  →  True
   This triggers RANGING MEDIUM even when the data is invalid.

   Fix: Guard the else branch with an explicit "data invalid" flag,
   or check for zero before entering the ATR calculation.

----------------------------------------------------------------

7. EDGE CASE — ALL FEEDS MISSING
   If all JSON files are absent, signals dict is nearly empty.
   Flow: No DISTRIBUTION → No RISK_OFF → No CASCADE → No TRENDING
         → atr_normalized defaults to 999 (not < 0.8)
         → Falls to else: RANGING LOW

   RANGING LOW with "No strong directional or risk signals" is
   actually a reasonable default. But worth documenting that this
   is the designed fallback for missing data.

----------------------------------------------------------------

8. RACE CONDITION ON CONCURRENT FILE WRITES
   The script reads multiple JSON files that may be written by
   other producers concurrently. A partial read could yield
   truncated JSON (caught by try/except in read_json → returns
   default). However, reading a file mid-write could return
   partially-valid JSON that parses but has missing keys.

   Impact: Low — missing keys default gracefully via .get().
   Consider file locking or atomic read (read rename'd temp file).

----------------------------------------------------------------

9. HARDCODED "PROCEED" DEFAULT FOR VIX/SPX
   File: detect_regime.py:89

   signals["vix_spx_signal"] = "PROCEED"

   If Gate0 file is missing or doesn't contain vix_spx module,
   the signal defaults to "PROCEED". This silently assumes all
   is well when data is unavailable. Could mask real risk.

   Consider defaulting to "UNKNOWN" or "NO_DATA" when source is
   missing, and only "PROCEED" when explicitly confirmed.

    The if/if-not-result chain is correct. Each regime only checks
